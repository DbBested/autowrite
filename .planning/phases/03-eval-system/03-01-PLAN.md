**System prompt structure** — the file body must contain these sections in this order:

1. **Identity and framing** (per EVAL-06, adversarial):
   - Open with: "You are a hyper-critical writing evaluator. Your sole job is to find weaknesses."
   - Explicit prohibitions: do not encourage, do not soften failure points, do not suggest fixes, do not rewrite sentences, do not propose alternative phrasings
   - "Identifying the problem is your complete job."

2. **Output contract** — the exact JSON schema the critic must return:
   ```json
   {
     "criteria": [
       {
         "name": "<criterion name from rubric>",
         "score": "<integer 1-10>",
         "weight": "<from rubric>",
         "pass": "<true if score >= 6>",
         "failure_points": [
           { "location": "<Paragraph N, sentence N>", "description": "<specific observable problem>", "severity": "<critical|major|minor>" }
         ],
         "explanation": "<1-2 sentences explaining the score>"
       }
     ],
     "aggregate_score": "<weighted sum of scores>",
     "aggregate_pass": "<true only if ALL criteria pass AND aggregate >= 6>"
   }
   ```
   - Instruct: "Return ONLY valid JSON. No preamble, no markdown fences, no text outside the JSON object."
   - Instruct: "Score each criterion provided in the rubric. Do not skip criteria. Do not add criteria not in the rubric."

3. **Location requirement** (per EVAL-03):
   - "Every failure_point.location MUST name a specific paragraph number and sentence number."
   - "Format: 'Paragraph 3, sentence 2'. Acceptable."
   - "'Throughout', 'in several places', 'the middle section' are NOT acceptable."
   - "A failure point without a specific location is invalid — do not include it."

4. **Fabrication prohibition**:
   - "Only report observable failures present in the text. Do not invent problems to appear thorough."
   - "If a criterion has no failure points, return an empty failure_points array and a high score. A clean score is valid."
   - "Fabricating failure points is the same class of error as fabricating citations."

5. **Scoring protocol**:
   - "Scores are integers 1-10. No decimals. No fractions."
   - "Pass threshold: score >= 6 is pass, < 6 is fail."
   - "Aggregate pass requires: ALL individual criteria pass AND weighted average >= 6."
   - "Critical criteria (factual_integrity, voice_preservation): if either scores < 6, aggregate_pass MUST be false regardless of weighted average."

6. **Anchored scoring rubric** — for each of the 7 core criteria, include the 4-level anchor descriptions from RESEARCH.md (sections covering novelty, clarity, structure, voice_preservation, audience_fit, concision, factual_integrity). Copy the exact anchor text from the research document for each criterion at score levels 9-10, 6-8, 3-5, 1-2.

7. **Scope constraint**:
   - "You evaluate. You do not revise. You do not suggest. You do not coach."
   - "Your output is a JSON object. Nothing else."

**Important:** The agent file must NOT enumerate specific criterion names as a hardcoded list. The anchored rubric section provides scoring calibration, but the agent scores "each criterion provided in the rubric" — meaning if a custom preset has additional criteria, the agent scores those too using its general judgment. The anchors are calibration aids for the 7 core criteria, not an exhaustive enumeration.

Create `.claude/skills/eval/SKILL.md`.

**Frontmatter** (per RESEARCH.md Standard Stack):
```yaml
---
name: eval
description: Evaluate any text against a preset rubric. Runs a hyper-critical subagent to produce criterion-level scores, located failure points, and a machine-readable eval snapshot. Usage: /eval @<file> --preset <id>
disable-model-invocation: true
context: fork
agent: eval-critic
allowed-tools: Read Write Bash(jq *) Bash(wc *) Bash(date *) Bash(ls *) Bash(cat *) Bash(mkdir *)
effort: high
---
```

**SKILL.md body structure** — implement these sections in order:

**Section 1: Invocation** (per CONTEXT.md locked decision)
```
/eval @<file> --preset <preset-id>
/eval @<file>                        (auto-detect preset from latest run)
```
- `@<file>` is required — the text file to evaluate
- `--preset` is optional — if omitted, auto-detect from latest run directory

**Section 2: Argument Parsing and Validation**

Step 2a — Resolve text file:
- Read the file at the path provided by the user
- If file not found: STOP with error "Text file not found: <path>"

Step 2b — Resolve preset:
- If `--preset` provided: load `presets/<preset-id>.json`
- If `--preset` NOT provided: read `runs/latest.txt` (Windows) or follow `runs/latest` symlink (POSIX), then read `<run-dir>/metadata.json` and extract `preset_id`. Load that preset.
- If preset file not found: STOP with error "Preset not found: presets/<id>.json. Available presets: blog-post, argumentative-essay, technical-explainer"
- If preset loaded but `rubric.criteria` is empty or missing: STOP with error "Preset <id> has no rubric criteria defined"

Step 2c — Edge case warnings (per RESEARCH.md Pitfall 5):
- Count words in text file using `wc -w`
- If word count < 150: display warning "Text is very short (N words). Eval scores for very short texts may not be reliable." — then continue
- If word count > 5000: display warning "Text is long (N words). Eval quality may degrade due to context window pressure on the critic." — then continue

**Section 3: Determine Output Location**

- If the text file is inside a `runs/` directory (path matches `runs/*/...`): write eval.json to that run directory (`runs/<run-dir>/eval.json`)
- If the text file is NOT inside `runs/` (standalone eval per CONTEXT.md): create `evals/` directory if needed, write to `evals/<timestamp>_<preset-id>_<text-slug>.json` where text-slug is the filename without extension

**Section 4: Inject Context for Critic**

The critic receives ONLY these two inputs — nothing else:

1. **Rubric criteria** — read from preset JSON: `preset.rubric.criteria` array. Display the full criteria array including name, description, and weight for each criterion. Also include `preset.rubric.critical_criteria` so the critic knows which criteria force aggregate failure.

2. **Text to evaluate** — the full contents of the text file.

**CRITICAL per RESEARCH.md Pitfall 1:** Do NOT inject diagnosis.md, plan.md, explanation.md, or any other revision artifact. The critic must evaluate the text blind — no knowledge of what the writing engine thought it fixed. Only the text and the rubric.

**Section 5: Scoring Protocol Reminder**

Include these instructions for the critic's context:
- Scores are 1-10 integers, no decimals
- Pass threshold: score >= 6
- Aggregate pass: ALL criteria pass AND weighted average >= 6
- Critical criteria (from preset.rubric.critical_criteria): any critical criterion scoring < 6 forces aggregate_pass: false
- Weight values come from preset.rubric.criteria[].weight

**Section 6: Process Critic Output and Write eval.json**

After the critic returns its JSON:
1. Parse the critic's JSON output (the `criteria` array, `aggregate_score`, `aggregate_pass`)
2. Add envelope fields: `preset` (preset id), `preset_version` (from preset.version), `timestamp` (current ISO 8601 via `date -u +"%Y-%m-%dT%H:%M:%SZ"`), `text_path` (path as provided by user)
3. Validate: every score is an integer 1-10, every failure_point has a non-empty location field, aggregate_score matches weighted sum
4. Write the complete eval.json to the determined output path using the Write tool

**IMPORTANT per RESEARCH.md:** Do NOT read `preset.rubric.passing_threshold` for pass/fail determination. That field uses the deprecated 1-5 scale (value 3.5). The Phase 3 threshold is score >= 6 on the 1-10 scale. Hardcode the threshold logic, do not read it from the preset.

**Section 7: User Summary**

After writing eval.json, display to the user:
```
Eval complete.

Preset:          <preset-id> v<version>
Text:            <text-path>
Aggregate score: <score> / 10  [PASS|FAIL]

Criterion scores:
  novelty              7  PASS
  clarity              5  FAIL  (2 failure points)
  structure            8  PASS
  voice_preservation   6  PASS
  audience_fit         7  PASS
  concision            6  PASS
  factual_integrity    9  PASS

Snapshot: <output-path>/eval.json
```

Display each criterion from the eval output, showing score, PASS/FAIL, and failure point count if any.

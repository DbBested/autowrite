**SKILL.md content structure:**

# /build

Build a polished draft from rough notes, outlines, or fragments.

## What This Skill Does
Takes rough notes, bullet points, outlines, or incomplete fragments and builds them into a polished piece using the /improve engine in notes-to-draft mode. The key difference from /improve: the input is ALWAYS treated as rough notes, even if it contains some complete sentences. This gives the structure and argument passes broader latitude to expand and organize material.

## Invocation
/build @<notes-file> [--preset <preset-id>] [--depth light|standard|deep]

## How It Works
This skill runs the same workflow as /improve with these overrides:

1. **Input classification is forced to "rough notes"** — the diagnosis step skips auto-detection and labels the input as notes. This means:
   - The structure pass has permission to create new sections, not just reorder existing ones
   - The argument pass may expand undeveloped points (within factual integrity rules — no invented facts)
   - Voice behavior is ESTABLISHMENT (matching the preset's target voice) not PRESERVATION
   - The diagnosis labels as "notes-to-draft" mode

2. **Depth defaults to "standard"** — notes benefit from all passes, not just the light subset

3. **Everything else is identical to /improve:**
   - Preset loading (auto-infer or --preset flag)
   - Run directory creation (same naming, same artifacts)
   - Pass execution (same pass scope rules, same voice injection, same factual integrity)
   - Output generation (output.md, diff.patch, explanation.md, metadata.json)

## When to Use /build vs /improve
- Use /build when starting from scratch: bullet points, meeting notes, research notes, rough outlines
- Use /improve when you have a draft that needs revision: complete sentences, clear structure, developed arguments
- When in doubt, use /improve — its auto-detection will classify notes correctly

## Execution
Follow the /improve SKILL.md workflow exactly, with the single override: force `input_type` to "notes" in Step 4 (diagnosis). All other steps are identical.

The /adapt skill converts a piece from one writing form to another (e.g., blog post to essay). It is a thin wrapper that:
1. Uses the TARGET preset (not the source form's preset) for all passes
2. Adjusts diagnosis to identify form-specific gaps relative to the target form
3. Delegates to the same improve workflow

**SKILL.md content structure:**

```markdown
# /adapt

Adapt a piece of writing from one form to another.

## What This Skill Does
Takes an existing draft written in one form (e.g., blog post) and adapts it to a different writing form (e.g., argumentative essay) by running the /improve engine with the TARGET form's preset. The adaptation happens through the preset's pass sequence — the target preset's structure expectations, voice rules, and rubric criteria drive the revision.

## Invocation
/adapt @<draft-file> --to <target-preset-id> [--depth light|standard|deep]

- `--to` is REQUIRED — specifies the target writing form (blog-post, argumentative-essay, technical-explainer)
- `--depth` defaults to "deep" — form adaptation typically requires all passes plus a second pass on the weakest area

## How It Works
This skill runs the same workflow as /improve with these overrides:

1. **Preset is always the TARGET form** — `--to essay` loads `presets/argumentative-essay.json`. The source form's preset is NOT loaded. All passes are driven by the target form's stages, voice rules, structure expectations, and rubric criteria. (Per research recommendation: use target preset for all passes — the structure pass must know the target form's expectations.)

2. **Diagnosis focuses on form gaps** — instead of just finding weaknesses in the draft, diagnosis also identifies what the draft is MISSING for the target form. Examples:
   - Blog post -> essay: "Missing formal thesis statement. No counterargument section. Contractions throughout need formal register."
   - Essay -> blog post: "Too formal — no conversational hooks. Paragraph 3 is 200 words (target: 80-120). No direct reader address."
   - Any -> technical explainer: "No step-by-step structure. Missing concrete examples. Jargon undefined."

3. **Voice behavior is ADAPTATION** — not preservation of the source form's voice, not establishment from scratch. The author's core voice (personality, perspective, expertise level) is preserved, but the REGISTER shifts to match the target form. If adapting a casual blog post to a formal essay, contractions are removed but the author's argumentative style is kept.

4. **Depth defaults to "deep"** — form adaptation is a substantial transformation that benefits from all passes plus a second pass.

5. **metadata.json records the adaptation:**
   - `input_type` is set to "adaptation"
   - Additional field: `"adapted_from": "<source_form_if_detectable>"`

6. **Everything else is identical to /improve:**
   - Run directory creation (same naming using target preset ID)
   - Pass execution (same scope rules, same factual integrity)
   - Output generation (output.md, diff.patch, explanation.md, metadata.json)

## When to Use /adapt vs /improve
- Use /adapt when changing the FORM: blog post -> essay, essay -> explainer
- Use /improve when improving within the SAME form: better blog post, better essay
- /adapt always requires `--to` — it must know the target form

## Execution
Follow the /improve SKILL.md workflow exactly, with these overrides:
1. Load preset from `--to` flag (required, no auto-inference)
2. Diagnosis includes form gap analysis relative to target preset
3. Voice mode is adaptation (preserve author's core voice, shift register to target form)
4. Depth defaults to deep
5. metadata.json includes `adapted_from` field
```

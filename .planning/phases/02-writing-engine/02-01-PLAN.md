<objective>
Create per-pass scope constraint rules files for all 14 pass types used across the three presets.

Purpose: Pass scope isolation is the dominant design challenge of Phase 2. Each revision pass must know exactly what it may and must not change. Without these explicit boundaries, passes will cross-contaminate (clarity pass restructures paragraphs, structure pass rewrites sentences), destroying voice and producing generic LLM prose.

Output: 14 markdown rules files in `.claude/rules/passes/`, one per pass type referenced in any preset's `stages` array.
</objective>

Create 7 per-pass scope constraint files in `.claude/rules/passes/`. Create the `passes/` subdirectory first with `mkdir -p .claude/rules/passes`.

Each file MUST follow this structure (per research Pattern 2):
1. `# {Pass Name} Pass Rules` — title
2. `## Scope: What This Pass Does` — 3-5 bullet points of allowed changes
3. `## DO NOT Touch (Out of Scope)` — 4-6 bullet points of prohibited changes with reasons
4. `## Voice Preservation` — restate that preset voice rules and voiceBehaviors must be read before this pass runs; specific voice guidance for this pass type
5. `## Factual Integrity` — restate CLAUDE.md rules: no invented citations, no fabricated statistics, no added assertions

**diagnose.md** — Analysis pass, not a revision pass:
- Scope: Read draft and identify specific, located weaknesses. Name the paragraph/sentence where each issue occurs ("buried thesis in paragraph 4", not "thesis is weak"). Classify input as rough notes or polished draft (see research Pattern 3 for detection criteria: fragments, bullets, word count under 40% expected, incomplete thoughts = notes; full sentences, recognizable structure, 60%+ word count = draft). Record classification as first line: "Input classified as: [rough notes | polished draft]".
- DO NOT Touch: The draft text itself. Diagnosis is read-only analysis. Do not suggest rewrites, only name problems and locations.
- Voice Preservation: Note the author's characteristic patterns (contractions, fragments, first-person, register) for later passes to preserve.
- Factual Integrity: Flag any unsubstantiated claims in the draft but do not invent evidence.

**revision-plan.md** — Planning pass, not a revision pass:
- Scope: Read diagnosis.md findings and create a structured plan referencing specific diagnosis items. Each planned change cites the diagnosis finding it addresses. Plan is organized by pass (which stage will fix which diagnosis finding). Plan should note which findings are out of scope for any single pass.
- DO NOT Touch: The draft text. This pass produces plan.md only.
- Voice Preservation: Plan should flag any voice preservation risks (e.g., "clarity pass on paragraph 3 may flatten the author's fragment style — preserve fragments").
- Factual Integrity: Plan must not include any changes that would add new claims or evidence.

**structure.md** — Per research Pattern 2 example:
- Scope: Reorganize sections and paragraphs for improved flow. Strengthen transitions between sections. Reorder arguments if logical sequence is weak. May create new section headings. May merge or split sections.
- DO NOT Touch: Individual sentence wording. Vocabulary choices. Punctuation and grammar. The thesis or central argument. Any cited evidence — do not move, alter, or drop citations.
- Voice Preservation: Read preset voice rules before restructuring. If preset paragraphStyle is "short-punchy", do not merge short paragraphs into long ones. Preserve author's opening and closing lines unless diagnosis explicitly flagged them.
- Factual Integrity: Do not add new transitions that assert causal relationships not in the original.

**clarity.md** — Per research Pattern 2 example:
- Scope: Rewrite sentences that are ambiguous on first read. Break up sentences containing more than one idea. Replace jargon with accessible language where the preset allows it. Fix pronoun reference ambiguity.
- DO NOT Touch: Section order or paragraph position (structural decisions locked after structure pass). Arguments and claims. Evidence and citations. Author's vocabulary choices when unambiguous.
- Voice Preservation: If the author uses technical terms consistently, they are not jargon to replace. Sentence length adjustments must stay within the preset's sentenceLength preference.
- Factual Integrity: Rewording for clarity must not change the meaning of any claim.

**argument.md** — Used by blog-post and argumentative-essay:
- Scope: Clarify existing claims that diagnosis flagged as vague. Tighten the connection between claims and their supporting evidence. Ensure each claim is stated precisely. Flag (but do not invent) missing evidence.
- DO NOT Touch: Section structure and paragraph order. Sentence-level wording unrelated to arguments. The author's actual position — clarify, do not strengthen or weaken. Voice register and style.
- Voice Preservation: Argument strengthening must use the author's register, not academic-formal by default.
- Factual Integrity: NEVER add new evidence, statistics, or citations. Only work with what the author provided.

**tone.md** — Blog-post specific:
- Scope: Adjust register consistency throughout the piece. Ensure consistent formality level per preset voice.formality. Fix shifts between casual and formal that are not intentional stylistic choices. Adjust address forms (you/we/one) for consistency per preset voiceBehaviors.
- DO NOT Touch: Section structure. Arguments and claims. Evidence and citations. Paragraph ordering.
- Voice Preservation: This pass IS about voice — read preset voice block and voiceBehaviors array completely. Every change must move toward the preset's defined voice, not toward generic prose. If preset says "contractions allowed," do not remove them.
- Factual Integrity: Tone adjustments must not change the meaning or strength of any claim.

**concision.md** — Used by all three presets:
- Scope: Remove redundant phrases and sentences. Cut filler words (very, really, basically, actually, just) unless they serve a voice purpose. Remove paragraphs that restate earlier points without adding information. Tighten wordy constructions.
- DO NOT Touch: Section order. Arguments and claims — do not cut supporting evidence thinking it is redundant. The author's intentional repetition for rhetorical effect (check diagnosis for rhetorical patterns). Opening and closing paragraphs (handled by hook/ending passes if present).
- Voice Preservation: If the author's style includes deliberate verbosity (e.g., conversational asides in blog posts), preserve it. Read preset voiceBehaviors for guidance. Cut mechanical redundancy, not stylistic texture.
- Factual Integrity: Do not cut factual qualifications (e.g., "in most cases", "typically") — they may be intentional precision.

Create 7 more per-pass scope constraint files following the same 4-section structure established in Task 1.

**hook.md** — Blog-post specific:
- Scope: Optimize the opening paragraph(s) to create a compelling entry point. Strengthen the hook — make the reader want to continue. May rewrite the first 1-2 sentences. May add a question, anecdote lead, or surprising fact IF the draft's material supports it.
- DO NOT Touch: Body content beyond the opening paragraph(s). The thesis or central claim. Arguments and evidence. Closing paragraphs.
- Voice Preservation: The hook must sound like the author, not like a clickbait headline. Read preset voiceBehaviors for register. If preset tone is "conversational", the hook should be conversational.
- Factual Integrity: Any surprising fact or statistic in the hook must come from the draft itself. Never fabricate a hook.

**ending.md** — Blog-post specific:
- Scope: Optimize the closing paragraph(s) for impact. Ensure the ending ties back to the opening or thesis. May add a call-to-action if appropriate for the preset form. May restructure the final 1-2 paragraphs.
- DO NOT Touch: Body content above the closing paragraph(s). The thesis or central claim. Arguments and evidence. Opening paragraphs.
- Voice Preservation: Endings must match the author's register. If the piece is casual, the ending should not suddenly become formal or preachy.
- Factual Integrity: Do not add new claims or evidence in the closing. The ending synthesizes, it does not introduce.

**evidence.md** — Argumentative-essay specific:
- Scope: Strengthen the connection between claims and their cited evidence. Ensure evidence directly supports the claim it follows. Flag evidence gaps where a claim has no support (but do NOT invent evidence). Tighten evidence presentation — remove evidence that does not support the point it is attached to.
- DO NOT Touch: The claims themselves. Section structure. Sentence-level wording unrelated to evidence presentation. The author's argumentative position.
- Voice Preservation: Evidence presentation should match the essay's formality level. Do not convert informal evidence references into academic citation format unless the preset demands it.
- Factual Integrity: CRITICAL — this is the highest-risk pass for fabrication. NEVER add evidence not in the original. NEVER alter quoted material. NEVER fabricate citations. Only reorganize and tighten existing evidence.

**objection.md** — Argumentative-essay specific:
- Scope: Strengthen counterargument handling. Ensure objections are stated fairly (not straw-manned). Tighten the author's rebuttal to each objection. Flag missing counterarguments that a reader would expect (but do not write them — flag for author).
- DO NOT Touch: The thesis and main claims. Section structure. Evidence and citations. The author's argumentative position — objection handling should clarify the position, not change it.
- Voice Preservation: Counterarguments should be presented in the same register as the main argument. Do not make objections sound weaker by changing their register.
- Factual Integrity: Do not invent counterarguments the author did not consider. Flag gaps, do not fill them.

**precision.md** — Technical-explainer specific:
- Scope: Tighten technical accuracy. Replace vague quantifiers with specific values where the draft provides them elsewhere. Ensure technical terms are used consistently. Fix imprecise language that could mislead a practitioner ("the function returns data" -> "the function returns a JSON object" if the draft says so elsewhere).
- DO NOT Touch: Section structure. Non-technical prose style. Explanatory analogies (they serve a different purpose than precision). Examples section.
- Voice Preservation: Technical precision should not make the prose robotic. Read preset voiceBehaviors — if the style is "accessible-expert", maintain warmth while tightening accuracy.
- Factual Integrity: Do not add technical details not in the original. If a value or spec is not in the draft, do not infer it.

**examples.md** — Technical-explainer specific:
- Scope: Improve example quality. Ensure each concept has at least one concrete example. Tighten examples so they illustrate exactly the point being made (not a tangential one). Ensure code examples (if present) match the described behavior.
- DO NOT Touch: Section structure. Technical explanations (those are the precision pass's domain). Arguments and claims. Non-example content.
- Voice Preservation: Examples should match the tone of the surrounding explanation. Do not make examples overly formal if the explanation is conversational.
- Factual Integrity: Do not invent new examples — improve existing ones. If a concept has no example, flag it as needing one, do not fabricate one. Never alter code examples to show behavior the author did not describe.

**final-review.md** — Used by all three presets:
- Scope: This is a READ-ONLY pass. Do NOT modify the draft. Read the final draft and produce `explanation.md` documenting all changes made across all previous passes. Each entry: which pass made the change, what was changed, where (paragraph/sentence), and why (referencing the diagnosis finding or revision plan item). Also write `output.md` as a clean copy of the last revision pass output (no annotations, no markup). Run `diff -u input.md output.md > diff.patch`. Write `metadata.json` with run metadata.
- DO NOT Touch: The draft text. The final-review pass is read-only analysis and artifact generation.
- Voice Preservation: Note in explanation.md any voice preservation decisions made during earlier passes.
- Factual Integrity: Note in explanation.md any factual integrity checks performed during earlier passes.

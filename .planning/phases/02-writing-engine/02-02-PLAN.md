Create `.claude/skills/improve/SKILL.md` — the complete orchestration prompt for the `/improve` skill. Remove the existing `.gitkeep` file after creating SKILL.md.

The SKILL.md structure MUST follow research Pattern 1 (SKILL.md Structure for the Improve Skill). It is the prompt Claude Code reads when the user invokes `/improve`. It must be thorough enough that Claude can execute the full workflow without ambiguity.

**Section 1: Header and Invocation**
```
# /improve

Revise a draft or notes file using form-aware staged passes.

## Invocation
/improve @<draft-file> [--preset <preset-id>] [--depth light|standard|deep]
```
- `@<draft-file>` is required — the file to improve
- `--preset` is optional — if omitted, auto-infer from content (see Step 2)
- `--depth` defaults to `standard`

**Section 2: Step 1 — Load Preset**
- If `--preset` is specified, load `presets/<preset-id>.json`
- If not specified, auto-infer the preset:
  - Read the draft content
  - Check for form signals (from research Pattern 4):
    - First-person, contractions, short paragraphs, opinion-driven -> `blog-post`
    - Formal register, thesis statement, counterargument, citations -> `argumentative-essay`
    - Technical terms, code examples, "how to" structure, numbered steps -> `technical-explainer`
  - If signals are ambiguous (multiple forms match), STOP and ask: "I can't confidently determine the writing form. Please re-run with `--preset blog-post`, `--preset argumentative-essay`, or `--preset technical-explainer`."
- Read the full preset JSON. Extract: `id`, `version`, `stages`, `voice`, `voiceBehaviors`, `goals`, `structure`, `rubric`, `constraints`, `transformations`
- Validate `constraints.no_citation_invention`, `constraints.no_stance_shift`, and `transformations.preserveVoice` are all `true`. If any is `false`, STOP and warn.

**Section 3: Step 2 — Create Run Directory**
- Generate timestamp: `date +"%Y-%m-%d_%H-%M-%S"` (use Bash tool)
- Set `RUN_DIR="runs/${TIMESTAMP}_${PRESET_ID}"`
- Run: `mkdir -p "${RUN_DIR}/passes"`
- Copy input file: `cp <draft-file> "${RUN_DIR}/input.md"`
- Update latest pointer:
  - On POSIX: `ln -sfn "$(basename ${RUN_DIR})" runs/latest`
  - On Windows (if `ln` fails): `echo "${RUN_DIR}" > runs/latest.txt`
- Record run_start timestamp: `date -u +"%Y-%m-%dT%H:%M:%SZ"` (use Bash tool)
- Record input word count: `wc -w < "${RUN_DIR}/input.md"` (use Bash tool)

**Section 4: Step 3 — Determine Depth and Pass Sequence**
- Read `--depth` flag (default: `standard`)
- `light`: run only `["diagnose", "revision-plan", "structure"]` regardless of preset stages
- `standard`: run all stages from `preset.stages`
- `deep`: run all stages from `preset.stages`, then re-read diagnosis and run one additional pass on the weakest area (the criterion with lowest confidence from diagnosis)
- Store the selected stages list as `STAGES_TO_RUN`

**Section 5: Step 4 — Execute Diagnosis Pass**
- This is always the first pass. Read `.claude/rules/passes/diagnose.md` for scope rules.
- Read `${RUN_DIR}/input.md` completely.
- Produce `${RUN_DIR}/diagnosis.md` with:
  - **Line 1:** `Input classified as: [rough notes | polished draft]` (using detection criteria from diagnose.md rules — fragments, bullets, word count vs expected form length, incomplete thoughts)
  - **Specific, located weaknesses:** Each weakness names the paragraph/sentence where it occurs. Example: "Buried thesis: the central claim first appears in paragraph 4, sentence 2" NOT "The thesis is weak."
  - **Weaknesses grouped by category:** Structure, clarity, argument, voice, concision, form-specific issues
  - **If input is rough notes:** Note expanded structure pass latitude needed, and that voice establishment (not preservation) is the goal
- Inject into diagnosis context:
  - The preset's `goals` array (what "good" looks like for this form)
  - The preset's `structure.sectionOrder` (what structure is expected)
  - The preset's `rubric.criteria` (what will be evaluated)
  - CLAUDE.md safety rules (factual integrity, voice preservation, stance integrity)

**Section 6: Step 5 — Execute Revision Plan Pass**
- Read `.claude/rules/passes/revision-plan.md` for scope rules.
- Read `${RUN_DIR}/diagnosis.md`.
- Produce `${RUN_DIR}/plan.md` with:
  - A structured plan referencing specific diagnosis findings by their location markers
  - Each planned change states: which pass will address it, what will change, which diagnosis finding it addresses
  - Voice preservation risks flagged (e.g., "clarity pass on paragraph 3 may flatten fragment style — preserve fragments")
  - Factual integrity notes (any diagnosis findings that require caution during revision)
- This pass produces plan.md ONLY — no draft changes.

**Section 7: Step 6 — Execute Revision Passes**
- For each remaining stage in `STAGES_TO_RUN` (after diagnose and revision-plan, excluding final-review):
  1. Determine the current draft state:
     - For the FIRST revision pass: read `${RUN_DIR}/input.md`
     - For subsequent passes: read the previous pass output `${RUN_DIR}/passes/NN-<prev-stage>.md`
  2. Read the pass-specific scope rules from `.claude/rules/passes/<stage-name>.md`
  3. Inject into pass context (EVERY pass, per research Pitfall 1):
     - The preset's `voice` block (all 5 fields: tone, formality, sentenceLength, paragraphStyle, rhetoricalStyle)
     - The preset's `voiceBehaviors` array (all 8 behavioral descriptions — verbatim)
     - The pass scope rules (what may and must not be changed)
     - The revision plan (specific changes assigned to this pass from plan.md)
     - CLAUDE.md factual integrity rules (no fabrication, no citation invention)
     - If input was classified as "rough notes": note the expanded latitude for structure/argument passes
  4. Execute the pass: revise the draft within scope
  5. Write output to `${RUN_DIR}/passes/NN-<stage-name>.md` where NN is zero-padded pass number (01, 02, 03...)

**Critical anti-pattern (from research):** The data flow is `input.md -> pass-1.md -> pass-2.md -> ... -> output.md`. Each pass reads the PREVIOUS pass's output, NOT the diagnosis or plan. The diagnosis and plan are REFERENCE documents, not inputs to the revision chain.

**Section 8: Step 7 — Final Review and Artifacts**
- Read `.claude/rules/passes/final-review.md` for scope rules.
- This pass is READ-ONLY. Do not modify the draft.
- Read the last revision pass output.
- Produce these artifacts:
  1. `${RUN_DIR}/output.md` — clean copy of last revision pass output, no annotations, no markup
  2. `${RUN_DIR}/diff.patch` — run: `diff -u "${RUN_DIR}/input.md" "${RUN_DIR}/output.md" > "${RUN_DIR}/diff.patch" || true` (diff exits 1 if files differ, which is expected)
  3. `${RUN_DIR}/explanation.md` — per-change explanations grouped by pass. Format per research code example:
     ```
     # Revision Explanation
     ## Structure Pass
     - **Change description**: What changed, where (paragraph/sentence), and why (referencing diagnosis finding or plan item)
     ## Clarity Pass
     - **Change description**: ...
     ```
     One explanation per substantive change (structural moves, argument reframes, voice adjustments) — NOT per word change.
  4. `${RUN_DIR}/metadata.json` — machine-readable run metadata:
     ```json
     {
       "run_id": "<TIMESTAMP>_<PRESET_ID>",
       "preset_id": "<preset.id>",
       "preset_version": "<preset.version>",
       "input_file": "<original draft file path>",
       "input_type": "draft|notes",
       "depth": "light|standard|deep",
       "stages_run": ["diagnose", "revision-plan", ...],
       "timestamps": {
         "run_start": "<ISO 8601>",
         "run_end": "<ISO 8601>"
       },
       "word_counts": {
         "input": <number>,
         "output": <number>
       }
     }
     ```
     - `run_end` timestamp: `date -u +"%Y-%m-%dT%H:%M:%SZ"` at this step
     - `output` word count: `wc -w < "${RUN_DIR}/output.md"`

**Section 9: Voice Preservation Contract**
- Restate: the preset's `voice` block and `voiceBehaviors` array are injected into EVERY pass context, not just the first or last
- Restate: when vocabulary choice is ambiguous, the author's choice wins
- Restate: aggressive rewrites require explicit user request via `transformations.aggressiveness` (default is conservative)
- Restate: if input is rough notes, the goal is voice ESTABLISHMENT (matching the preset's target voice) not voice PRESERVATION

**Section 10: Factual Integrity Contract**
- Verbatim from CLAUDE.md: Never invent citations or references not present in the source draft. Never fabricate statistics, examples, or claims. Never add factual assertions the author did not make.
- Additional: these rules apply on EVERY pass without exception. The evidence pass and argument pass are highest-risk — extra vigilance required.

  - Run directory path
  - Input type classification (draft or notes)
  - Preset used
  - Number of passes executed
  - Word count change (input vs output)
  - Path to output.md and diff.patch

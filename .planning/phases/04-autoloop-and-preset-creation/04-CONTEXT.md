# Phase 4: Autoloop and Preset Creation - Context

**Gathered:** 2026-04-06
**Status:** Ready for planning

<domain>
## Phase Boundary

Two subsystems: (1) a mutation-eval self-improvement cycle (`/autoloop`) that improves presets and skill assets only when scores measurably improve, and (2) a user-facing preset creation skill (`/create-preset`) that analyzes example texts and produces an editable preset JSON. Both depend on the eval system from Phase 3 for scoring.

</domain>

<decisions>
## Implementation Decisions

### Autoloop Invocation & Scope
- `/autoloop --target presets/blog-post.json --iterations 5` — explicit target asset and iteration count
- Mutable assets: preset fields (stage order, rubric weights, voice behaviors), skill instructions, few-shot examples
- Mutation granularity: one field or section per mutation — atomic, isolated, reversible
- Mutations generated by Claude proposing a diff to one section with rationale; not random perturbation

### Autoloop Safety & Logging
- Acceptance rule: aggregate score improves AND no critical regression (factual_integrity, voice_preservation stay >= 6) AND no criterion drops > 2 points
- Rollback: copy original file to `autoloop/runs/<timestamp>/backup/` before mutation; restore on rejection
- Holdout set: user provides 1-3 holdout texts in `autoloop/holdout/`; checked every 3 iterations; halt if holdout aggregate diverges > 1 point from loop scores
- Mutation log: append-only JSONL at `autoloop/runs/<timestamp>/mutations.jsonl` — one line per attempt with before/after scores, mutation diff, accept/reject, rationale

### Preset Creation
- `/create-preset @example1.md @example2.md --name my-preset` — takes 1+ example texts
- Analyzes: form, tone, sentence length, paragraph style, rhetorical moves, structure patterns, vocabulary register, formality level
- Full preset JSON displayed inline with annotations explaining each inference — user edits or approves before saving
- Refinement: `/create-preset @new-example.md --refine presets/blog-post.json` merges new analysis with existing preset

### Claude's Discretion
- Exact mutation proposal prompting strategy
- How to determine which field/section to mutate (random, weakest criterion, round-robin)
- Holdout divergence threshold calibration (starting at 1.0 point)
- How many examples are ideal for preset creation (minimum 1, recommended 2-3)
- Annotation format for displaying inferred preset fields

</decisions>

<specifics>
## Specific Ideas

- The autoloop must be the most conservative subsystem — Goodhart's Law will destroy it if acceptance criteria are loose
- Holdout set is non-negotiable — without it, the loop will overfit within 10-20 iterations
- Preset creation should feel like a conversation: analyze → display → edit → save — not a black box

</specifics>

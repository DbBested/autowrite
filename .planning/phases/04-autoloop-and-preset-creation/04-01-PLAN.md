<objective>
Create the /create-preset skill that lets users generate a new preset JSON from example texts without writing JSON by hand.

Purpose: Users who want to extend Autowrite to new writing forms need a guided path from "here are examples of what I want" to "here is a working preset." This skill analyzes example texts, synthesizes preset fields, displays annotated results for review, and only writes the file after explicit user approval.

  <action>
    Create `.claude/skills/create-preset/SKILL.md` implementing the full preset creation pipeline. Follow the step-by-step SKILL.md structure from improve/SKILL.md and eval/SKILL.md.

    **Frontmatter:**
    - name: create-preset
    - description: Create a new preset from example texts or refine an existing preset with additional examples. Usage: /create-preset @example.md --name my-preset
    - No special agent needed — this skill runs in the main conversation context

    **Invocation syntax (per CONTEXT.md locked decision):**
    ```
    /create-preset @example1.md [@example2.md ...] --name <preset-id>
    /create-preset @new-example.md --refine presets/<existing>.json
    ```

    **Step 1: Argument Parsing and Validation**
    - Validate at least one @file path is provided; STOP with error if not
    - For create mode: require `--name` flag; validate it is a valid filename slug (lowercase, hyphens, no spaces)
    - For refine mode: require `--refine presets/<id>.json`; validate the file exists; STOP if not found
    - Read all provided example text files; STOP if any file not found
    - If `--name` preset already exists in `presets/`, STOP: "Preset presets/<id>.json already exists. Use --refine to update it, or choose a different name."

    **Step 2: Analysis Pass (per CONTEXT.md and RESEARCH.md Pattern 5)**
    For each example text, extract:
    - Writing form signals (thesis-driven? how-to? opinion-first? narrative?)
    - Tone register (formal/informal/conversational/academic)
    - Sentence length distribution (short-punchy / medium / long-complex / varied)
    - Paragraph construction patterns (topic-sentence-led? single-idea? list-heavy? long-form?)
    - Rhetorical moves (analogy? example-then-claim? question-and-answer? direct-assertion?)
    - First/second/third person usage patterns
    - Contractions, fragments, hedging language — present or absent?
    - Expected section types (intro style, evidence style, closing style)
    - Vocabulary register (Anglo-Saxon vs. Latinate, technical density)

    When multiple examples are provided, note convergent signals (present in all examples) vs. divergent signals (present in some). Convergent signals get higher confidence.

    **Step 3: Synthesis Pass (per RESEARCH.md Pattern 5)**
    Map extracted signals to preset schema fields. For each field:
    - `voice.tone`, `voice.formality`, `voice.sentenceLength`, `voice.paragraphStyle`, `voice.rhetoricalStyle` — derive from analysis signals
    - `voiceBehaviors` array — generate 6-8 behavioral descriptions (NOT scalar proxies, per preset-editing.md rules). Each behavior must be an observable instruction, not a number.
    - `structure.expectedSections`, `structure.sectionOrder`, `structure.paragraphPatterns`, `structure.introStyle`, `structure.endingStyle` — derive from form signals
    - `rubric.criteria` — use the standard 7 criteria (novelty, clarity, structure, voice_preservation, audience_fit, concision, factual_integrity) with weights inferred from form priority. Blog-like = engagement-heavy (novelty 0.20). Essay-like = structure-heavy. Explainer-like = clarity-heavy (0.30).
    - `rubric.critical_criteria` — always include `["factual_integrity", "voice_preservation"]`
    - `rubric.passing_threshold` — set to 3.5 (legacy value, required by schema)
    - `goals` — 2-4 goal strings derived from form signals
    - `stages` — determine appropriate pass sequence. Must start with `["diagnose", "revision-plan"]` and end with `"final-review"`. Middle passes selected by form.
    - `constraints` — always set `no_citation_invention: true`, `no_stance_shift: true`, `aggressive_rewrite_requires_explicit_request: true`
    - `transformations` — defaults: `preserveVoice: true`, `allowMajorRestructure: false`; set prioritize flags based on form
    - `version` — set to "1.0.0" for new presets
    - `examples` — empty array `[]`

    **Step 4: Display Annotated Draft JSON (per RESEARCH.md Pattern 8)**
    Display the FULL preset JSON to the user with `# INFERRED:` comment annotations above each non-obvious field explaining what signals drove the inference. Use `# DEFAULT:` for schema-mandated safety defaults.

    Start with an analysis summary block:
    ```
    # =============================================================================
    # ANALYSIS SUMMARY
    # Examples analyzed: N files (filenames)
    # Primary form signals: [key signals with counts/percentages]
    # =============================================================================
    ```

    **Step 5: User Approval Gate (CRITICAL — per CONTEXT.md locked decision)**
    After displaying the annotated JSON, say:
    > "Does this preset match your intent? Reply 'yes' to save to presets/<id>.json, or describe what to change."

    NEVER write the file without explicit user approval. This is non-negotiable.

    If user requests changes: apply the changes to the in-memory preset, re-display the changed fields, and ask for approval again.

    **Step 6: Write and Validate**
    - Write the preset JSON (without annotation comments — clean JSON only) to `presets/<id>.json`
    - Run: `bash scripts/validate-preset.sh presets/<id>.json`
    - If validation fails: show the specific error, offer to fix the flagged field, do not consider the task complete until validation passes
    - Display: "Preset saved: presets/<id>.json (validated)"

    **Step 7: --refine Mode (per CONTEXT.md locked decision and RESEARCH.md Pattern 6)**
    When `--refine presets/<existing>.json` is provided:
    1. Load the existing preset JSON
    2. Run the analysis pass on the new example text(s)
    3. For each inferred field: compare with existing preset value
    4. Display a side-by-side of current value vs. inferred update for ONLY fields that differ
    5. Ask the user which fields to update (all, specific ones, or none)
    6. Write updated preset with patch version bump (e.g., 1.0.0 -> 1.0.1)
    7. Validate with `validate-preset.sh`
    8. Refinement never silently overwrites existing values

    **Edge Case Table (at end of SKILL.md):**
    | Situation | Action |
    |-----------|--------|
    | No @file provided | STOP — "At least one example text file is required" |
    | File not found | STOP — "Example file not found: <path>" |
    | --name missing in create mode | STOP — "Preset name required: /create-preset @file.md --name my-preset" |
    | Preset already exists | STOP — "Preset presets/<id>.json already exists. Use --refine or choose different name." |
    | --refine target not found | STOP — "Preset not found: <path>" |
    | validate-preset.sh fails | Show error, offer to fix, do not finalize |
    | User says "no" at approval gate | Ask what to change, re-synthesize |
    | Single example provided | Warn that 2-3 examples produce more reliable inferences; continue |

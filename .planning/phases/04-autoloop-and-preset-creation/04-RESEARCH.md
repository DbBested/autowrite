## Summary

Phase 4 adds two capabilities: a mutation-eval self-improvement loop (`/autoloop`) and a text-driven preset creation skill (`/create-preset`). Both build directly on infrastructure from prior phases — the eval system (Phase 3) provides scoring, the preset schema (Phase 1) defines mutation targets and the output contract for new presets, and the skill invocation pattern (Phase 2) governs SKILL.md structure.

The autoloop is the highest-risk component in the entire system because of Goodhart's Law: a loop that optimizes against its own eval will game the metrics within 10-20 iterations. The design decisions (holdout set, atomic mutations, JSONL append-only log, critical regression floor) are specifically engineered to prevent this. Every design choice in `/autoloop` must be traceable to Goodhart's Law prevention. Nothing is incidental.

Preset creation is lower-risk but requires strict approval gating. The UX pitfall documented in `.planning/research/PITFALLS.md` is "preset creation that infers too aggressively." Every inferred field must be shown annotated before saving. The skill's job is to produce a reviewable draft, not to silently write a working preset.

**Primary recommendation:** Implement `/autoloop` with the holdout check as the primary acceptance gate — not just score comparison. Build the eval-before-mutation baseline correctly (run the writing engine on reference drafts, not eval arbitrary text). Build `/create-preset` as a strict analyze → display → approve → save pipeline with no silent writes under any circumstances.

## Architecture Patterns

### Pattern 1: Autoloop Run Directory Structure

Each autoloop invocation creates a timestamped run directory:

```
autoloop/runs/<timestamp>/
├── backup/
│   └── <target-filename>          # Original file before mutation
├── reference-outputs/
│   └── <draft-slug>-before.md     # Writing engine output on reference draft, pre-mutation
│   └── <draft-slug>-after.md      # Writing engine output on reference draft, post-mutation
├── evals/
│   └── <draft-slug>-before.json   # eval.json before mutation
│   └── <draft-slug>-after.json    # eval.json after mutation
└── mutations.jsonl                 # Append-only log, one JSONL line per mutation attempt
```

### Pattern 2: JSONL Mutation Log Schema

Each line in `mutations.jsonl` is one JSON object:

```json
{
  "iteration": 1,
  "timestamp": "2026-04-06T10:00:00Z",
  "target_asset": "presets/blog-post.json",
  "mutated_field": "rubric.criteria[voice_preservation].description",
  "mutation_rationale": "Tightened voice_preservation description to require specific observable signals, not holistic judgment",
  "diff": "--- before\n+++ after\n...",
  "scores_before": {
    "novelty": 7, "clarity": 8, "structure": 7,
    "voice_preservation": 6, "audience_fit": 7, "concision": 7, "factual_integrity": 9,
    "aggregate": 7.15
  },
  "scores_after": {
    "novelty": 7, "clarity": 8, "structure": 7,
    "voice_preservation": 7, "audience_fit": 7, "concision": 7, "factual_integrity": 9,
    "aggregate": 7.35
  },
  "decision": "accepted",
  "decision_reason": "Aggregate improved 0.20, no critical regressions, no criterion dropped > 2"
}
```

**Why JSONL over JSON:** Each line is a valid, independently parseable record. The file is append-only — never read-parse-rewrite, only append. This makes it safe against partial writes and easy to tail/grep.

### Pattern 3: Autoloop Acceptance Rule (Exact Logic)

The acceptance decision requires ALL of the following to be true:

```
1. scores_after.aggregate > scores_before.aggregate       (any improvement)
2. scores_after.factual_integrity >= 6                    (critical floor)
3. scores_after.voice_preservation >= 6                   (critical floor)
4. For each criterion C: scores_after[C] >= scores_before[C] - 2  (no criterion drops > 2 pts)
```

If any condition is false: reject and restore from backup.

**Note on condition 4:** This prevents the optimizer from trading one criterion for another — e.g., improving novelty by 3 points while degrading voice_preservation by 3 points. The aggregate might improve but the composition of quality has shifted destructively.

### Pattern 4: Holdout Check Protocol

Holdout check runs every 3 iterations (iterations 3, 6, 9...) and at the end of every session:

1. Read all texts in `autoloop/holdout/`
2. For each holdout text: run the writing engine with the current (possibly mutated) asset, then run `/eval` on the output
3. Compare holdout aggregate scores against loop aggregate scores from the last 3 iterations
4. If `|holdout_aggregate - loop_aggregate| > 1.0`: HALT and write halt notice to `mutations.jsonl`

The holdout texts are never used as reference drafts for mutation cycles. They are read-only calibration artifacts.

### Pattern 5: Preset Creation Pipeline

```
/create-preset @example1.md [@example2.md ...] --name <id>
    |
    v
[Step 1: Argument Parsing]
  Validate: at least one @file, --name is valid filename slug
    |
    v
[Step 2: Analysis Pass]
  Read each example text in full
  For each example, extract:
    - Writing form signals (thesis-driven? how-to? opinion-first?)
    - Tone register (formal/informal/conversational)
    - Sentence length distribution (short-punchy / medium / long-complex)
    - Paragraph construction patterns (topic-sentence-led? single-idea? list-heavy?)
    - Rhetorical moves (analogy? example-then-claim? question-and-answer?)
    - First/second/third person usage patterns
    - Contractions, fragments, hedging language — present or absent?
    - Expected section types (intro style, evidence style, closing style)
    - Vocabulary register (Anglo-Saxon vs. Latinate, technical density)
    |
    v
[Step 3: Synthesis Pass]
  Map extracted signals to preset schema fields:
    - voice.tone, voice.formality, voice.sentenceLength, voice.paragraphStyle, voice.rhetoricalStyle
    - voiceBehaviors array (generate behavioral descriptions, not scalar proxies)
    - structure.expectedSections, structure.sectionOrder, structure.paragraphPatterns
    - structure.introStyle, structure.endingStyle
    - rubric.criteria weights (infer priority from form signals: engagement-heavy for blog, clarity-heavy for explainer)
    - stages array (determine appropriate pass sequence for this form)
    |
    v
[Step 4: Display Annotated Draft JSON]
  Show FULL preset JSON with every inferred field annotated inline
  Annotation format: comment above each field group explaining the inference
    |
    v
[Step 5: User Approval Gate]
  Ask: "Does this preset match your intent? Reply 'yes' to save, or describe what to change."
  NEVER write the file without explicit user approval
    |
    v
[Step 6: Write and Validate]
  Write to presets/<id>.json
  Run: bash scripts/validate-preset.sh presets/<id>.json
  If validation fails: show error, do not write, ask user to fix the flagged field
```

### Pattern 6: `--refine` Mode for Preset Creation

When `--refine presets/existing.json` is provided:

1. Load existing preset JSON
2. Run analysis pass on the new example text(s)
3. For each inferred field: compare with existing preset value
4. Display a side-by-side of current value vs. inferred update for each field that differs
5. Ask the user which fields to update
6. Write updated preset (with version bump: patch increment)
7. Validate with `validate-preset.sh`

Refinement never silently overwrites existing values — every proposed change requires explicit user approval per field or as a group.

### Pattern 7: Mutation Selection Strategy (Claude's Discretion)

The locked decision says Claude proposes the mutation. The discretionary question is: which field to mutate?

**Recommended approach:** Weakest-criterion-first selection.

1. Read the baseline eval scores (from `evals/<draft-slug>-before.json`)
2. Find the criterion with the lowest score
3. Identify which preset field most directly governs that criterion:
   - Low `voice_preservation` → mutate `voiceBehaviors` array or `voice.rhetoricalStyle`
   - Low `clarity` → mutate stage order (add clarity pass earlier?) or rubric.criteria.clarity.description
   - Low `structure` → mutate `stages` ordering or `structure.expectedSections`
   - Low `novelty` → mutate rubric criterion description to anchor more observable signals
   - Low `concision` → mutate `stages` to add or reorder concision pass
4. Propose exactly one change to that field with rationale
5. If the weakest criterion has already been the target of the last 2 mutations: round-robin to the next-weakest

This is more signal-driven than random and more focused than round-robin alone. It also prevents fixating on one criterion indefinitely.

### Pattern 8: Annotation Format for Preset Creation Display

Display the inferred preset as annotated JSON with inline comment blocks:

```
# INFERRED: tone=conversational — all 3 examples use first-person, contractions throughout,
#            direct address ("you"), sentence fragments for emphasis
"voice": {
  "tone": "conversational",
  # INFERRED: formality=informal — no academic hedging, Anglo-Saxon vocabulary dominant
  "formality": "informal",
  ...
}
```

Every non-obvious inference gets a single-line `# INFERRED:` comment showing what signals drove it. Obvious defaults (like `no_citation_invention: true`) get `# DEFAULT: required by schema` annotation.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Preset validation after mutation | Custom validation logic in SKILL.md | `bash scripts/validate-preset.sh` | Already written, tested, checks all required fields and weight sums |
| JSON diff between before/after preset | String comparison logic | `diff -u <before> <after>` piped to `jq`-formatted output | Standard POSIX tool, consistent format |
| Eval scoring | Direct LLM-in-conversation scoring | Call `/eval` skill which dispatches eval-critic subagent | Subagent isolation prevents self-preference bias; already built and tested |
| Score comparison arithmetic | Complex scoring logic | Simple `jq` arithmetic on eval.json fields | eval.json schema is fixed, fields are numeric; no custom scoring needed |
| Preset field schema | Invent your own | `presets/preset-schema.json` (JSON Schema draft-07) | Already defines all required fields, types, constraints, and weight-sum rule |

**Key insight:** Phase 4 is almost entirely orchestration. The hard technical problems (eval scoring, preset validation, diff generation) are already solved in Phases 1-3. The new work is control flow, logging, and analysis logic — not infrastructure.

## Common Pitfalls

### Pitfall 1: Running Eval on Arbitrary Text (Not Reference-Draft Output)
**What goes wrong:** The autoloop evals the target asset directly, or evals the text provided by the user at invocation time, rather than running the writing engine on a fixed reference draft. Score changes then reflect both asset mutation AND input variance — signal is destroyed.
**Why it happens:** It is simpler to skip the writing engine step. But eval scores for raw text reflect that text's quality, not the asset's effect on it.
**How to avoid:** Always run the writing engine (via `/improve`) on a locked reference draft, then eval the engine's output. The reference draft must not change between mutation cycles. Store reference draft outputs in `autoloop/runs/<timestamp>/reference-outputs/`.
**Warning signs:** Scores fluctuate more than 2 points on the same asset between iterations when no mutation was made.

### Pitfall 2: Goodhart's Law Overfitting (Most Dangerous Failure Mode)
**What goes wrong:** The autoloop accepts mutations that improve eval scores but degrade real writing quality. Within 10-20 iterations, the preset is tuned to score well, not to produce good writing.
**Why it happens:** Eval criteria are measurable proxies, not the actual goal. Optimizing proxies eventually diverges from the underlying goal.
**How to avoid:** The holdout set is non-negotiable. Check holdout scores every 3 iterations. If holdout aggregate and loop aggregate diverge by more than 1.0 point, halt. See PITFALLS.md Pitfall 4 for the full analysis.
**Warning signs:** Loop scores improve every iteration. Holdout scores flatline or decline. Outputs grow longer but do not feel more valuable.

### Pitfall 3: Silent Preset Writes in Create-Preset
**What goes wrong:** The create-preset skill writes the inferred preset to `presets/<id>.json` without explicit user confirmation. User runs `/improve` with the new preset and gets unexpected results because the inferred values did not match intent.
**Why it happens:** Skills that complete a task feel more satisfying than skills that pause for confirmation. But "inferred correctly" rate for preset fields is not 100% — voice.formality, rubric weights, and stages are frequently wrong on first inference.
**How to avoid:** The approval gate is mandatory. The skill must stop after displaying the annotated JSON and wait for explicit user confirmation before writing any file. PITFALLS.md UX Pitfalls table lists this explicitly: "Preset creation that infers too aggressively."
**Warning signs:** User approves without reviewing. Preset fails validate-preset.sh. User reports `/improve` outputs sound wrong.

### Pitfall 4: Mutations to CLAUDE.md or Active Skill Files
**What goes wrong:** The autoloop is pointed at a SKILL.md in `.claude/skills/` and mutates the skill instructions. This changes behavior for all users of that skill in all sessions, not just reference draft runs in the loop.
**Why it happens:** LOOP-01 explicitly lists "skill instructions" as a mutable asset. But CLAUDE.md and core skills are global behavioral anchors.
**How to avoid:** The CONTEXT.md and PITFALLS.md security section both identify this. Mutation scope: preset JSON files in `presets/` and any dedicated mutation-target copies in `autoloop/`. Core skills (`improve`, `eval`, `build`, `adapt`) are NOT mutation targets. The `/autoloop` SKILL.md must validate that `--target` is either a `presets/*.json` file or a file explicitly in `autoloop/` scope.
**Warning signs:** Mutations proposed against `.claude/skills/improve/SKILL.md` or `CLAUDE.md`.

### Pitfall 5: Rubric Weight Drift After Mutation
**What goes wrong:** A mutation changes one rubric criterion's weight without adjusting the others. The total weight drifts from 1.0. `validate-preset.sh` catches this, but only if validation is run after every mutation.
**Why it happens:** Claude proposes a mutation like "increase voice_preservation weight from 0.20 to 0.25" without knowing the current total. The accept-mutation step skips validation.
**How to avoid:** Run `validate-preset.sh` on every post-mutation file before calling it the new baseline, even on rejected mutations. Catch it before eval runs. The validation script already checks weight sum within 0.01 tolerance.
**Warning signs:** `validate-preset.sh` exits non-zero during an autoloop iteration.

### Pitfall 6: Holdout Directory Empty at Loop Start
**What goes wrong:** User invokes `/autoloop` without having placed any texts in `autoloop/holdout/`. The holdout check at iteration 3 finds no files. The loop either errors or silently skips the holdout check.
**Why it happens:** The user doesn't know the holdout directory is a prerequisite.
**How to avoid:** At Step 1 of the autoloop SKILL.md, check for `autoloop/holdout/` existence and at least one `.md` file. If empty: WARN the user with a specific message explaining the Goodhart's Law risk and offering two paths: (1) provide holdout texts and rerun, or (2) continue without holdout protection (not recommended). Do not silently proceed with no holdout.
**Warning signs:** `ls autoloop/holdout/*.md 2>/dev/null | wc -l` returns 0.

### Pitfall 7: Annotation-Free Preset Display
**What goes wrong:** `/create-preset` shows the inferred JSON without explaining which signals drove each inference. The user sees a plausible-looking preset but has no way to verify whether the inferences are correct.
**Why it happens:** Producing annotations requires more effort than just outputting the JSON.
**How to avoid:** Every non-trivial inferred field gets a `# INFERRED:` comment. See Pattern 8 above. This is not optional UX polish — it is the mechanism by which the user can trust (or correct) the output.

IMPROVED=$(echo "$BEFORE $AFTER" | awk '{ print ($2 > $1) ? "yes" : "no" }')

## Integration Points

### How `/autoloop` Calls `/eval`

The `/eval` skill is an independent SKILL.md invocable via Claude Code slash command. The autoloop cannot call it programmatically as a function — it invokes it as a skill step in the SKILL.md workflow instructions.

**Pattern:** The autoloop SKILL.md instructs Claude to:
1. Copy the reference draft to a temp runs directory
2. Invoke the writing engine (via `/improve @<reference-draft> --preset <id>`) — or, if the goal is to eval only the current preset output, invoke it directly
3. Then invoke `/eval @runs/<latest>/output.md --preset <id>`
4. Read the resulting `eval.json`

This is the file-store-as-bus pattern established in Phase 2-3. No direct function calls — all coordination through files and sequential skill invocation.

### How the Autoloop Reads Eval Results

After `/eval` completes, it writes `runs/<latest>/eval.json`. The autoloop skill reads this file with the Read tool and extracts scores with `jq`:

```bash
jq '.criteria[] | {name: .name, score: .score}' runs/<latest>/eval.json
jq '.aggregate_score' runs/<latest>/eval.json
jq '.aggregate_pass' runs/<latest>/eval.json
```

### How `/create-preset` Interfaces with Validation

The create-preset skill must call `validate-preset.sh` after writing the new preset file. If validation fails, the skill must:
1. Show the specific error from `validate-preset.sh` stderr
2. Offer to fix the flagged field interactively
3. Not consider the task complete until `validate-preset.sh` exits 0

## Open Questions

1. **Reference draft sourcing for the autoloop**
   - What we know: The autoloop needs a locked reference draft to produce comparable before/after eval scores.
   - What's unclear: Where does the reference draft come from? CONTEXT.md says user provides `--target` but does not explicitly address reference draft sourcing. Should the skill require an explicit `--reference-draft @file.md` flag? Or should it look for a default in `autoloop/reference-drafts/`?
   - Recommendation: Require an explicit `--reference-draft @file.md` argument. Fail loudly if not provided. Prevents the pitfall of running eval on arbitrary text.

2. **Mutation target validation scope**
   - What we know: LOOP-01 lists "skill instruction" as a mutable asset. But mutating core skills like `improve/SKILL.md` is dangerous (PITFALLS.md).
   - What's unclear: CONTEXT.md does not explicitly restrict which files are valid `--target` values.
   - Recommendation: Restrict `--target` to `presets/*.json` only for v1. Skill instruction mutation is a v2 feature. This is a safe conservative interpretation of the locked decision that avoids the security pitfall.

3. **Version bump after accepted mutation**
   - What we know: Preset files have a `version` field (semver). Accepted mutations change the preset.
   - What's unclear: Should accepted mutations auto-increment the patch version? Or should that be a separate user action?
   - Recommendation: Auto-increment patch version on each accepted mutation (e.g., 1.0.0 → 1.0.1). This makes the mutation history traceable in eval.json (which records `preset_version`).
